History!

A thousand years ago...

Linus posted on a forum he is been working on a operating system, as a hobby. The community starts helping and giving suggestions. Months later, the Linux kernel releases its first version.
BitKeeper was a proprietary distributed VCS which had free accounts for open-source developers.
A git developer tried to reverse engineer a BitKeeper protocol, which was against their terms.

What do we need?

  • Speed
  • Simple design
  • Strong support for non-linear development
  • Fully distributed
  • Able to handle large projects
Although created to suit the Linux kernel needs, you can use Git for any files you want to version control.

Creating a repository

git init
Creates a .git directory which contains the git database

The three sections

Repository

  • Where Git stores the metadata and object database of the project
  • .git directory
  • Also known as HEAD
This is what makes Git fully distributable.

Working Directory

  • The actual files
  • Retrieved from the Repository and placed on the disk

Staging Area

  • Stores all changes that what will go in the next commit
  • Also known as Index

The real  face of a commit

A tree of commits

What are those 98ca9 92ec2 things?

The identifier and its integrity

Everything in Git is check-summed with a SHA-1 hash


A SHA-1 hash is string composed of 40 hexadecimal characters

24b9da6552252987aa493b52f8696cd6d3b00373


There is a total of 2160 hashes

or 1.4615016373309029182036848327163x1048

Branching

The concept

Branching is how you diverge from the main line of development and continue to do work without changing the main line

On Git

A branch is just a pointer to a commit

Creating and deleting branch is super fast.

Different from other VCS, which creates a copy of the entire project which may take long.

The master branch

  • The default branch
  • It's not special

Creating a branch

git branch branch-name

git branch testing

Switching branches

git checkout branch-name

git checkout testing

What if you commit now?


git commit -a -m "testing commit"

-a includes all files to the staging area

Going back to master


git checkout master

Cleans your working directory to make it look where master branch was.

Working on master again


git commit -a -m "master commit"

Branches are cheap!

Just a file with the 40-character SHA-1 hash of the commit

Use and abuse them!

Basic Merging

The situation

You are working on a web application which is used by thousands of users.


Today, you decided that you are going to work on Issue #53.

git checkout -b iss53


You include a commit with a partial fix.

git commit -a -m "Fix part of issue 53"

The repository

Boom!

Your boss calls you saying there is an issue affecting all users and you need to fix it immediately.


But I'm currently working on Issue #53 and already changed a lot of files. I don't want them with the fix.


Don't worry! With Git, this is easy peasy!

git checkout master

Let's fix it!

You create a new branch and fix the issue after some minutes of hard work.

git checkout -b hotfix

git commit -a -m "Fix the typo on the home page"

The current repository

Fix is ready! Start deployment!

After some testing, you now want to merge the fix to the master branch where the production code is in.

                        $ git checkout master
                        $ git merge hotfix
                        Updating f42c576..3a0874c
                        Fast-forward
                         index.html | 2 ++
                         1 file changed, 2 insertions(+)

A fast-forward merge

Let's go back to Issue #53

But first... Let's delete the hotfix branch, since you don't need it anymore.

                        $ git branch -d hotfix
                        Deleted branch hotfix (3a0874c)

Finishing Issue #53

You get back to work on the issue.

git checkout iss53

git commit -a -m "Fix the second part of issue 53"

The repository

It's time... to merge!

You decided that Issue #53 is now fixed and ready to be merged into master.

                        $ git checkout master
                        Switched to branch 'master'
                        $ git merge iss53
                        Merge made by the 'recursive' strategy.
                        index.html |    50 +
                        1 file changed, 50 insertions(+)

That's a little different...

Git is working behind the scenes!

Since Git stores pointers to the parent commit, he knows easily who is the Common Ancestor. Then he's able to do a 3-way merge.

A wild merge commit has appeared!

In 5 minutes!
